# 项目核心宪章 (Project Constitution)
<!-- 本宪章确立了项目的核心原则，旨在保证代码质量、测试标准、用户体验和性能的卓越，确保项目的长期流畅运行。 -->

## 核心原则 (Core Principles)

### I. 代码质量优先 (Code Quality First)
<!-- 致力于编写清晰、可维护且健壮的代码 -->
代码质量是项目的基石。所有代码必须遵循 SOLID 原则和 DRY（Don't Repeat Yourself）原则。代码的可读性优于“聪明的”实现。必须强制执行静态代码分析（Linting）和严格的代码审查流程。任何技术债务的引入都必须有明确的记录和偿还计划。

### II. 全面的测试标准 (Comprehensive Testing Standards)
<!-- 测试是交付信心的来源 -->
没有测试的代码是不完整的。我们实行多层级测试策略：单元测试覆盖核心逻辑，集成测试验证模块交互，端到端（E2E）测试保障关键业务流程。新功能的提交必须包含相应的测试用例，且必须通过自动化 CI/CD 流水线的验证。

### III. 用户体验一致性 (Consistent User Experience)
<!-- 提供无缝且愉悦的用户旅程 -->
用户体验（UX）的一致性至关重要。所有界面元素、交互模式和反馈机制必须遵循统一的设计系统和规范。无论用户身处应用的哪个部分，都应感受到熟悉且直观的操作体验。无障碍设计（Accessibility）是标准配置，而非可选功能。

### IV. 严格的性能要求 (Strict Performance Requirements)
<!-- 速度和响应性是核心竞争力 -->
性能不是事后优化的补充，而是设计之初的考量。页面加载时间（LCP）、交互延迟（FID/INP）和布局稳定性（CLS）必须满足 Core Web Vitals 的优秀标准。任何导致性能显著回退的变更都将被视为阻断性问题，禁止合并。

### V. 项目流畅性与持续交付 (Project Fluidity & Continuous Delivery)
<!-- 保持开发与发布的顺畅 -->
为了保证项目的流畅性，我们追求自动化的构建、测试和部署流程。开发环境应尽可能模拟生产环境。小步快跑，频繁集成，确保问题能被尽早发现和解决，从而保持开发节奏的连贯和发布的稳定性。

## 执行标准 (Execution Standards)

### 技术栈与工具链 (Tech Stack & Toolchain)
- **代码规范**：统一使用 ESLint/Prettier 等工具强制格式化。
- **测试框架**：统一测试框架，确保测试用例的可读性和可维护性。
- **性能监控**：集成性能监控工具，实时跟踪关键性能指标。

## 开发工作流 (Development Workflow)

### 质量门禁 (Quality Gates)
1.  **提交前**：本地运行 Lint 和单元测试。
2.  **代码审查**：至少一名其他开发者审查通过，关注逻辑、规范和设计模式。
3.  **合并前**：CI 流水线必须全绿（构建成功、测试通过、无性能回退）。

## 治理 (Governance)
本宪章高于其他开发惯例。对宪章的修改需要团队核心成员的讨论和批准。所有团队成员有义务监督和维护这些原则的执行。
